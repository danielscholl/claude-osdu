# Comment Tone Reference

MR comments generated by this plugin must sound like a developer wrote them. This reference
defines the voice.

## Rules

1. **No em dashes.** Use periods, commas, or rewrite the sentence.
2. **No opener praise.** Skip "Great work", "Nice changes", "Well done". Start with substance.
3. **No filler.** Cut "It's worth noting", "I noticed that", "Just wanted to mention".
4. **Short sentences.** If a sentence has two clauses joined by a dash, split it into two sentences.
5. **Paragraphs over bullets** when the content flows naturally. Bullets are fine for lists of
   distinct items (failed jobs, action items), not for narrating a review.
6. **Technical and factual.** State what you observed and what it means. Skip hedging.
7. **Proportionate length.** A simple MR gets 2-3 sentences. A complex one gets a few paragraphs.
   Never pad.

## Patterns to Avoid

| Instead of | Write |
|------------|-------|
| The fix looks solid — it correctly addresses the root cause | The fix correctly addresses the root cause. |
| It's worth noting that coverage dropped below 80% | Coverage dropped below 80%. |
| Great work! The implementation is clean and well-tested. | The implementation is clean and well-tested. |
| I noticed that the Azure provider now throws instead of returning null | The Azure provider now throws instead of returning null. |
| Just a thought — maybe we should add a test for the null case | Could use a test for the null case. |

## Calibration

Read the comment back and ask: would this look out of place in a GitLab thread full of
developer comments? If it sounds like a bot wrote it, rewrite it.
